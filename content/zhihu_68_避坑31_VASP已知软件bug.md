## Bug 1: VASP 6.5.0 ISPIN=2 + LNONCOLLINEAR = CRASH

**严重程度**：🔴 致命（直接crash）

**症状**：设置`ISPIN=2`和`LNONCOLLINEAR=.TRUE.`后，VASP在读取INCAR阶段直接segmentation fault。没有任何有意义报错。

**影响版本**：VASP 6.5.0（6.4.3及之前正常，6.5.1已修复）

**根因**：自旋极化+非共线磁性的内部数据结构冲突。VASP 6.5.0重构了磁性模块，但`ISPIN`和`LNONCOLLINEAR`的组合路径测试遗漏。

**绕路方案**：
1. 升级到VASP 6.5.1+
2. 如果被锁死在6.5.0：使用`LNONCOLLINEAR=.TRUE.` + 去掉`ISPIN`标签（非共线计算不需要单独设置ISPIN）
3. 如果计算本质上只需要共线磁性：`ISPIN=2` + `LNONCOLLINEAR=.FALSE.`（默认）

**审稿人视角**：使用6.5.0+LNONCOLLINEAR的论文，建议在方法段注明VASP版本号（精确到小版本），避免审稿人质疑可复现性。

## Bug 2: EOS fitting的数值不稳定性（VASP 6.x全系列）

**严重程度**：🟡 中等（结果可复现但数值有偏）

**症状**：用VASP 6.x做状态方程(EOS)拟合时，某些体积点的能量出现0.5-2 meV/atom的"跳变"。不是总能收敛问题——SCF已经收敛到1E-8，但能量和相邻体积点不连续。

**根因**：VASP 6.x的默认FFT网格分配算法在胞参数变化时会重新分配网格→不同体积点使用了"不同精度"的FFT→E-V曲线不光滑。这个问题在VASP 5.x中不存在。

**绕路方案**：
1. **强制固定FFT网格**：在INCAR中显式设置`NGXF/NGFY/NGZF`为一个固定的足够大的值（如2×默认），所有体积点使用相同的NGX
2. 使用VASP 5.4.4做EOS（如果有license）
3. 增大`PREC=Accurate` + `ENMAX=1.5×ENMAX`→减少但不会完全消除

**验证方法**：做EOS后检查E-V曲线的二阶导数平滑性。如果出现锯齿→FFT网格问题。

## Bug 3: DFPT声子计算的对称性bug（IBRION=5/6/7/8）

**严重程度**：🔴 严重（结果定性错误）

**症状**：用DFPT（IBRION=7/8）或有限位移法（IBRION=5/6）计算声子时，Γ点声学模不零（或显著偏差>1 cm⁻¹），或某些声学模频率为负值但结构已经充分优化。

**根因**：VASP的对称性分析在某些空间群+超胞组合下错误地识别了对称性不等价的原子为等价→力常数矩阵错误→声子频率错误。该bug在官方的"已知问题"列表中但修复不全。

**绕路方案**：
1. **关闭对称性**：`ISYM=0`→强制VASP不对体系施加任何对称性
2. 如果ISYM=0后声学模仍不零：检查结构优化是否充分（EDIFFG ≤ -0.001 eV/Å）
3. 用Phonopy+有限位移法（IBRION=6）交叉验证DFPT结果
4. **金标准**：有限位移法比DFPT更不容易受对称性bug影响

## Bug 4: LDAUTYPE默认值的隐藏变化

**严重程度**：🟡 中等（静默改变）

**症状**：同一INCAR在VASP 5.4和VASP 6.x中给出不同的DFT+U结果。不是泛函变了、不是U值变了——是`LDAUTYPE`的默认值变了。

**根因**：
- VASP 5.4: LDAUTYPE默认=2（Dudarev方法，Ueff=U-J）
- VASP 6.x: LDAUTYPE默认=2（表面相同），但内部实现的投影算符更精确→导致有效Ueff略有不同
- 某些版本中，LDAUTYPE=1（Liechtenstein方法）的行为被修改但文档没及时更新

**绕路方案**：
1. **永远显式设置LDAUTYPE**——不要依赖默认值
2. 如果比较5.4和6.x的结果→必须用相同的VASP版本+相同的INCAR
3. 方法段写清楚：`LDAUTYPE=2 (Dudarev approach)`——审稿人会感谢你

## Bug 5: WAVECAR的跨节点兼容性（HPC集群特有问题）

**严重程度**：🔴 严重（数据损坏）

**症状**：在集群的一个节点上生成的WAVECAR，复制到另一个节点后无法读取或导致VASP crash。错误信息："ERROR: WAVECAR header mismatch"或静默的segfault。

**根因**：不同节点的编译环境（MPI库/编译器版本/数学库）可能生成不同二进制格式的WAVECAR。这不是VASP核心的bug，但VASP不提供跨平台WAVECAR兼容性检查→静默数据损坏。

**绕路方案**：
1. **绝对不要在不同编译版本的VASP之间传输WAVECAR**
2. 如果必须跨节点：使用WAVEDER（文本格式）代替WAVECAR作为初始波函数
3. 保险做法：续算时用ISTART=0（从零开始）或ISTART=1（从CHGCAR生成波函数）而不是ISTART=2（从WAVECAR续算）
4. 检查集群的VASP module是否一致：`module list`确认所有计算节点使用同一版本的VASP

## Bug 6: 大体系（>500原子）的并行I/O瓶颈→静默hang

**严重程度**：🟡 中等（不会crash但无限等待）

**症状**：体系>500原子时，VASP在"reading WAVECAR"阶段hang（不动了）。不crash、不报错、不算SCF——就是卡住了。Top显示进程在100% CPU但什么都不做。

**根因**：VASP的MPI I/O实现在大体系+多节点时存在死锁条件。所有核尝试同时读写同一个大文件→I/O拥塞→部分核超时等待→无限hang。

**绕路方案**：
1. `LWAVE=.FALSE.` →不写WAVECAR
2. `LCHARG=.FALSE.` →不写CHGCAR（如果需要电荷密度，用LCHARG=.TRUE.但接受I/O开销）
3. 使用`NCORE`和`KPAR`优化并行：大体系推荐NCORE≈每个节点的核心数，KPAR=节点数
4. 如果必须写大文件：使用lustre等并行文件系统（不是NFS）

## Bug 7: VASP 6.x的"能量漂移"——静默的不收敛

**严重程度**：🟡 中等（不易发现但影响所有结果）

**症状**：电子步SCF收敛到EDIFF，但离子步之间能量不单调下降。结构优化跑了几十步后，能量反而比初始结构高。EDIFF和EDIFFG都设得够严，但问题依旧。

**根因**：VASP 6.x的电荷密度混合方案（IMIX/AMIX/BMIX）在某些体系中（特别是d电子体系）的默认值不够保守→电荷密度在离子步之间有"记忆效应"→前一步的密度污染下一步的SCF→能量不单调。

**绕路方案**：
1. **每次离子步重新开始电荷密度**：`ICHARG=2`（从叠加原子密度开始，不用上一步密度）
2. 更保守的混合：`AMIX=0.1; BMIX=0.0001; AMIX_MAG=0.2; BMIX_MAG=0.0001`
3. 如果是d/f电子体系：`LMAXMIX=4`（d电子）或`LMAXMIX=6`（f电子）
4. 终极方案：`ALGO=All`（Davidson算法，更稳健但更慢）

## 总结

VASP是一个极其强大但充满隐藏陷阱的工具。这7个已知bug中，至少3个（Bug 1/3/5）可能导致结果定性错误。最好的防御策略是：
